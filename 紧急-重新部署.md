# 🚨 紧急：重新部署（文件上传 404 问题）

## ❌ 问题分析

**现象：**
```
[文件上传] 上传成功 {statusCode: 404}
"path":"/api/files/upload"
```

**原因：**
- 部署时使用了 `backend` 目录，而不是 `cloud-deploy` 目录
- `backend/Dockerfile` 从源码构建，但源码**可能不是最新的**
- `cloud-deploy/Dockerfile` 使用预构建的 JAR（包含最新代码）

**证据：**
- 其他接口（/api/idols, /api/diaries）正常 ✅
- 文件上传接口 404 ❌
- 说明部署了旧版本代码（没有 FileController）

## 🔍 两个 Dockerfile 的区别

### backend/Dockerfile（旧的，有问题）

```dockerfile
# 从源码构建
FROM maven:3.9.6-eclipse-temurin-17 AS builder
COPY pom.xml .
COPY src ./src
RUN mvn package -DskipTests -B  # ← 这里构建的源码可能不是最新的
```

**问题：**
- 依赖 backend 目录中的源码
- 如果源码不是最新的，构建出的 JAR 也是旧的
- 构建时间长（需要下载依赖、编译）

---

### cloud-deploy/Dockerfile（新的，正确）

```dockerfile
# 直接使用预构建的 JAR
FROM eclipse-temurin:17-jre-jammy
COPY app.jar /app/app.jar  # ← 使用本地已经构建好的 JAR（最新的）
```

**优点：**
- 使用确定是最新的 JAR 包（2026-06-08 12:04 构建）
- 构建速度快（< 1 分钟）
- 不依赖源码版本

## 🆘 如果还是 404

### 1. 确认上传的文件夹

在云托管构建日志中查看：
```
#14 [stage-1 3/4] COPY --from=builder /app/target/idol-diary-backend-1.0.0.jar app.jar
```

如果看到这行，说明使用了旧的 Dockerfile（从源码构建）

应该看到：
```
COPY app.jar /app/app.jar
```

### 2. 检查 JAR 包内容

```bash
# 本地验证 cloud-deploy/app.jar 包含 FileController
jar -tf cloud-deploy/app.jar | findstr FileController
```

应该看到：
```
BOOT-INF/classes/com/idoldiary/controller/FileController.class
```

### 3. 重新构建 JAR

如果 `cloud-deploy/app.jar` 确实是旧的：

```cmd
cd d:\SOFT\爱豆时光日记\backend

# 清理并重新构建
mvn clean package -DskipTests

# 复制最新的 JAR
copy target\idol-diary-backend-1.0.0.jar ..\cloud-deploy\app.jar

# 重新部署 cloud-deploy 文件夹
```
